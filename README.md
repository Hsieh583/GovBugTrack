# GovBugTrack (GBT) - 台灣公共治理缺陷追蹤系統

> **「當法律變成演算法，而執法變成自動化腳本時，公民就是最後的 Debugger。」**

## 📌 專案願景 (Vision)
本專案將「台灣政府機關」視為一組**公共服務供應商**。我們發現目前的行政系統（如交通檢舉、鄰里管理）在執行過程中，經常出現邏輯設計錯誤（標線陷阱）、資源分配失衡（官僚自動化開單）等系統性缺陷（Bugs）。

本專案旨在提供一套**結構化的「缺陷報告與稽核 SOP」**，協助非政治專業的民眾以最低的溝通成本與心理負擔，參與公共事務的修正與監督。

---

## 🛠 核心運作邏輯 (Methodology)

我們不採取情緒化的抗爭，而是採用**「工程思維」**介入：
1. **結構化陳述**：將民怨轉化為 `Issue Report`（案由、事實、法規、訴求）。
2. **合規性審查**：以政府宣稱遵循的《行政程序法》與 ISO 標準為準繩進行外部稽核。
3. **低成本介入**：提供模組化模板，讓民眾能像「開單給廠商」一樣快速完成陳情。

## 📂 資料夾結構 (Repository Map)

* `docs/Standard_Operating_Procedures/`：
    * `L1_Micro_Adjustment.md` - 微型介入（如：標誌不明、垃圾收運）。
    * `L2_Systemic_Audit.md` - 系統審查（如：惡意檢舉成案、區域停車規劃）。
* `templates/Bug_Reports/`：
    * `Traffic_Design_Fault.md` - 道路設計陷阱報告模板。
    * `Malicious_Reporting_Appeal.md` - 惡意檢舉救濟申請模板。
* `knowledge_base/`：
    * `API_Legal_Reference.md` - 常用行政法規「白話接口說明」。
    * `Precedents_Library.md` - 成功撤銷處分的判決先例庫。

## 🚀 快速入門 (Quick Start)

### 第一步：定義 Bug 類型
當你遇到不合理的行政執法（例如：在標示不明處被檢舉「未打方向燈」）：
1. 停止憤怒，開啟本專案。
2. 下載 `templates/Bug_Reports/Traffic_Design_Fault.md`。

### 第二步：填寫 Bug Report
* **環境描述**：拍下現場標線、紀錄檢舉時間點。
* **缺陷分析**：引用本專案提供的「合規性原則」（如比例原則）。

### 第三步：提交與追蹤
* 透過「市長信箱」或「議員窗口」送出。
* 將回覆內容節錄至本專案，進行「版本控管」與「結案追蹤」。
