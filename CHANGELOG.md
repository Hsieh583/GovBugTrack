# 改版歷史 (Changelog)

## v2.0 - 2026年2月8日 🚀

### 重大升級：「老主任秘訣」系統整合

本次改版引入經驗豐富的公民行動者總結的實戰技巧，大幅提升申訴成功率。核心理念：**從「情緒化抗議」升級到「策略性施壓」**。

---

### 新增功能

#### 1️⃣ 成本分析模組 (Cost-Benefit Analysis)
**檔案位置**：`Templates/Cost_Benefit_Analysis.md`

**功能說明**：
- 用「行政成本」說服主管，而非只談法律
- 提供判決字號查找指南
- 計算敗訴機率與人力成本
- 給對方一個「理性選擇」的下台階

**核心話術**：
> 「若本案進入行政訴訟，貴機關需指派法制人員出庭約 20-30 小時，且依據判決字號 XXX，敗訴機率約為 65%。為節省社會資源與行政成本，建請於本階段進行撤銷或改善。」

**適用時機**：
- ✅ 初次申訴遭駁回後的複查階段
- ✅ 準備提起訴願或行政訴訟前
- ❌ 不建議在初次申訴時使用（過於強硬）

**預期效果**：讓主管自己算出「撤單」比「對抗」更符合行政效益。

---

#### 2️⃣ 民意槓桿自動生成器 (Press Release Draft)
**檔案位置**：`Templates/Press_Release_Draft.md`

**功能說明**：
- 當「文攻（公文）」無效時啟動「武嚇（輿論）」
- 提供 4 種不同平台的範本：
  - 📱 臉書/爆料公社貼文（簡潔型）
  - 📰 新聞稿（正式型）
  - 📊 議員陳情懶人包（圖文包）
  - ⭐ Google 評論/1999 App 負評（快速戰術）

**核心策略**：
- 將複雜的法律問題簡化成「秒懂圖卡」
- 用對比圖展示矛盾（駕駛人視角 vs Google 鳥瞰圖）
- 數據化呈現（「80% 車輛未打方向燈」比「很多人」更有力）

**使用原則**：
- ⚠️ 這是「最後手段」，不要一開始就用
- ✅ 先走完正式申訴流程，給機關改正機會
- ✅ 確保內容真實，避免誹謗風險

**預期效果**：
- 官僚最怕不合理的政策被「攤在陽光下」
- 給機關壓力：「這個案子拖下去會變成新聞」
- 給議員誘因：「這個案子有媒體價值」

---

#### 3️⃣ 協作分工策略指南 (Collective Action Strategy)
**檔案位置**：`Guides/Collective_Action_Strategy.md`

**功能說明**：
- 當問題具有「公共性」時，組織集體行動
- 提供三種協作模式：
  - **模式 A**：同一問題，多角度證據（飽和攻擊）
  - **模式 B**：不同管道，同一訴求（多管齊下）
  - **模式 C**：分階段升級（壓力遞增）

**核心理念**：一人一信不如一人一題

**操作方式**：
- 由發起人統一製作「核心訴求文件」
- 參與者提供不同角度的證據（不同時段、不同視角）
- 統一行動時間窗口（同一週內，但不同天）
- 避免被視為「重複陳情」或「組織性騷擾」

**預期效果**：
- 讓個案問題變成「系統性問題」
- 迫使機關必須開專案會議處理
- 分散個人風險，無法針對特定人

**成功案例**：
- 新北市某路段標線改善（15 人集體陳情，1 個月內改善）
- 高雄市工業區違停執法爭議（30 位業者陳情，成功調整為夜間開放停車）

**三大實戰技巧**：

1. **數據勝過形容詞**
   - ❌ 失敗：「很多人受害」
   - ✅ 成功：「30 分鐘內 50 輛車，40 輛（80%）未打方向燈」

2. **給出替代路徑**
   - ❌ 失敗：「這裡不能開單」
   - ✅ 成功：「建議將此處紅線改為『20:00 後開放停車』之黃線」

3. **保留回覆版本，建立版控紀錄**
   - 所有機關回函都是未來訴訟或質詢的證據
   - 建議用 Git 管理申訴歷程

### 設計理念

#### 為什麼要加入「老主任秘訣」？

1. **從防禦到主動**：原版系統偏向「防禦性參與」，新版加入「策略性施壓」能力

2. **從個人到集體**：原版只教個人申訴，新版教你如何組織集體行動

3. **從文攻到武嚇**：原版只有公文戰，新版加入輿論戰工具（但強調這是最後手段）

4. **從法律到成本**：官僚不怕你講法律（他們有法制人員），但怕你算成本（因為主管要看效率）

#### 使用流程升級

```
v1.0 流程：
  正式申訴 → 若駁回則放棄或訴訟

v2.0 流程：
  正式申訴
    ↓（若駁回）
  加入成本分析（理性施壓）
    ↓（若仍無回應）
  準備民意槓桿素材（輿論威脅）
    ↓（先私下給議員/記者）
  評估 1-2 週
    ↓（若仍無改善）
  公開發布 + 集體行動
```

### 使用建議

#### 新手使用路徑

1. **第一次申訴**：使用 `Traffic_Design_Complaint.md` 等基礎模板
2. **申訴被駁回**：加入 `Cost_Benefit_Analysis.md` 模組
3. **需要支援**：發起 GitHub Issue，參考 `Collective_Action_Strategy.md`
4. **準備升級**：研究 `Press_Release_Draft.md`，但先不公開

#### 進階使用路徑

1. **建立 Git 專案管理你的案件**：
   ```
   MyCase/
   ├── initial_complaint.md
   ├── response_1.pdf
   ├── appeal_with_cost_analysis.md
   ├── response_2.pdf
   ├── press_release_draft.md (備用)
   └── evidence/
   ```

2. **參與或發起集體行動**：
   - 在 GitHub Issues 搜尋相同問題
   - 若無人發起，自己建立 Issue
   - 使用 `Collective_Action_Strategy.md` 指南

3. **貢獻成功案例**：
   - 案例整理（去識別化）
   - 分享有效話術
   - 回報改善結果

### 風險提醒

#### 新增工具的使用風險

**成本分析模組**：
- ⚠️ 不要虛構判決字號（可被查證）
- ⚠️ 敗訴率估算要有依據
- ✅ 語氣保持專業，避免威脅性

**民意槓桿生成器**：
- ⚠️ 確保內容真實，避免誹謗
- ⚠️ 不要攻擊特定承辦人
- ✅ 目的是「促進改善」而非「個人報復」

**協作分工策略**：
- ⚠️ 避免變成「騷擾」（100 人同一天送相同內容）
- ⚠️ 保護參與者隱私（使用代號）
- ⚠️ 避免被政治勢力利用
- ✅ 堅持「就事論事」，只談問題本身
